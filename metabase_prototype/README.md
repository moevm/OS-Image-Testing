## Быстрый старт

Скопируйте файл с переменными окружения и при необходимости поменяйте переменные:

```
cp .env.example .env
```

Запустите контейнеры командой:

```
docker compose up -d
```

Примечание: metabase нужно некоторое время загрузиться после старта контейнера, обычно не больше минуты.

## Контейнеры и их взаимодействие

- **Контейнер основной базы данных** – `mvp-postgres` содержит синтетические данные, в будущем будет удален, а metabase будет подключаться напрямую к базе данных проекта
  - Внутренний порт: `5432`.
  - Внешний порт на хосте: задаётся переменной `POSTGRES_PORT` (по умолчанию `5432`).
  - Metabase подключается к этой базе по внутреннему имени сервиса `postgres` и порту `5432`. (Подключение настраивается внутри metabase)

- **Контейнер служебной базы метаданных** – `mvp-metabase-meta-db` хранит внутренние настройки Metabase: вопросы, дашборды, информацию о подключениях.
  - Внутренний порт: `5432`.
  - Контейнер `metabase` подключается к нему по имени `metabase-meta-db` и порту `5432`.

- **Контейнер самого Metabase** – `mvp-metabase` запускает веб‑сервер Metabase.
  - Внутренний порт: `3000`.
  - Внешний порт на хосте: задаётся переменной `METABASE_PORT` (по умолчанию `3000`).
  - Подключается к служебной базе `metabase-meta-db` для хранения своих метаданных.
  - После настройки пользователем подключается также к основной базе (`postgres`) для работы с данными.

## Переменные окружения

**Для основной базы данных (`postgres`)**

- `POSTGRES_USER=admin` – имя пользователя для подключения к основной БД.
- `POSTGRES_PASSWORD=admin` – пароль пользователя.
- `POSTGRES_DB=metabase_app` – имя основной БД.
- `MVP_RESULTS_DB=mvp_results` – дополнительная БД, создаётся внутри того же контейнера (используется скриптами инициализации).
- `POSTGRES_PORT=5432` – порт, который открывается на хосте для доступа к основной БД.
- `POSTGRES_DATA_VOLUME=postgres_data` – имя docker‑тома для хранения данных основной БД.

**Для служебной базы метаданных Metabase**

- `METABASE_META_DB_NAME=metabase` – имя БД, где metabase хранит свои настройки.
- `METABASE_META_USER=metabase` – пользователь служебной БД.
- `METABASE_META_PASS=metabase` – пароль пользователя.
- `METABASE_META_VOLUME=metabase_meta_data` – имя тома для данных служебной БД.

**Для контейнера Metabase**

- `METABASE_IMAGE=metabase/metabase:latest` – используемый образ Metabase.
- `METABASE_PORT=3000` – порт на хосте для доступа к веб‑интерфейсу.
- `METABASE_APP_VOLUME=metabase_app` – том для хранения файлов приложения Metabase.


## Ожидаемые форматы входных данных (BMF-подобные)

### Формат 1 – для скалярных данных

В этом формате каждый измеренный параметр представлен отдельным полем внутри объекта. Поля могут быть сгруппированы по смысловым категориям (например, `job1_read`, `job1_cpu`).

**Структура**

- Корневой объект содержит произвольное количество категорий (например, `"job1_read"`, `"job1_cpu"`).
- Каждая категория – это объект, где ключи – названия метрик (например, `"io_bytes"`, `"lat_ns"`, `"usr_cpu"`).
- Каждая метрика – это объект с полями:
  - `value` (обязательное) – числовое значение метрики.
  - `lower_value` (опциональное) – нижняя граница доверительного интервала.
  - `upper_value` (опциональное) – верхняя граница доверительного интервала.

**Пример**

```json
{
  "job1_read": {
    "io_bytes": {"value": 10485760},
    "lat_ns": {"value": 6000, "lower_value": 3000, "upper_value": 15000}
  },
  "job1_cpu": {
    "usr_cpu": {"value": 12.5}
  }
}
```

### Формат 2 – для векторных данных

Этот формат используется для временных рядов или многократных измерений одной метрики. Все точки должны иметь одинаковые единицы измерения.

**Структура**

Корневой объект содержит следующие поля:

- `benchmark` (строка, обязательное) – имя бенчмарка (например, `"fio"`, `"sysbench"`).
- `metric` (строка, обязательное) – название метрики (например, `"throughput_mb_s"`, `"latency_us"`).
- `points` (массив, обязательное) – список измерений, каждое измерение – объект с полями:
  - `value` (число, обязательное) – значение метрики.
  - `lower_value` (число, опциональное) – нижняя граница.
  - `upper_value` (число, опциональное) – верхняя граница.

**Пример**

```json
{
  "benchmark": "fio",
  "metric": "throughput_mb_s",
  "series": "read",
  "unit": "MB/s",
  "sample_interval_sec": 10,
  "points": [
    {"value": 123.4, "lower_value": 121.0, "upper_value": 126.5},
    {"value": 125.1, "lower_value": 122.3, "upper_value": 128.0}
  ]
}
```
