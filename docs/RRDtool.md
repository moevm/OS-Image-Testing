# RRDtool

[RRDtool](https://www.rrdtool.org/rrdtool/doc/rrdtool.en.html) (Round Robin Database Tool) — это набор утилит и библиотека для логирования и визуализации временных рядов. Основная идея — хранить метрики так, чтобы размер базы оставался постоянным во времени благодаря кольцевому формату хранилища.

RRDtool привязан ко многим языкам: Perl, Python, Ruby и т.д.

## Как устроено хранилище

RRDtool хранит временные ряды. Например:
- Сетевой трафик (байты/бит в секунду по интерфейсу)
- Загрузка процессора
- Использование памяти
- Счётчики запросов
- Температура, уровень шума, энергопотребление
- Количество посетителей и любые другие числовые показатели

### Ограничения:
- Данные должны быть числами (целыми или дробными)
- Каждое значение привязывается ко времени

### Какие типы метрик доступны
- GAUGE - хранит фактическое значение, без вычисления скорости изменения. Например: температура, текущий объём памяти
- COUNTER - Используется для монотонно растущих счётчиков. Передается накопительный счётчик, а RRDtool сохраняет скорость роста: `(текущее - прошлое) / время`. Счётчик не должен уменьшаться (кроме переполнения), а переполнение 32/64 бит RRDtool обрабатывает автоматически.
- DERIVE - то же самое, что COUNTER, но без обработки переполнения и с поддержкой отрицательной скорости. Например, изменение свободного места на диске
- ABSOLUTE - передается, сколько произошло с прошлого измерения. RRDtool делит это на значение шага и хранит как среднюю скорость за интервал
- DCOUNTER - аналог COUNTER для вещественных чисел, когда счётчик растёт не на целые значения. RRDtool хранит скорость, но учитывает, что счётчик может быть только либо растущим, либо убывающим, но не тем и другим одновременно
- DDERIVE - аналог DERIVE для вещественных чисел: хранится скорость изменения, которая может быть отрицательной, но без проверки переполнения
- COMPUTE - это тип источника данных, у которого значения не задаются напрямую: каждое значение автоматически вычисляется по формуле из других источников данных, уже сохранённых в том же файле базы данных RRDtool

### Как устроено хранилище RRDtool
RRDtool хранит данные в бинарных файлах с расширением `.rrd`. Это кольцевая база данных: размер файла задаётся при создании и остаётся постоянным.

При создании файла резервируется фиксированное число ячеек для хранения данных. Эти ячейки можно представить как точки на окружности. Есть указатель на текущую ячейку: при записи новых данных он двигается вперёд по кругу и, когда возвращается в начало, новые значения начинают перезаписывать самые старые.

### Где RRDtool хранит данные

RRDtool не использует отдельный сервер базы данных — все данные хранятся в обычных
файлах с расширением `.rrd` в файловой системе.

Каждый файл `.rrd` содержит:
  - заголовок с описанием шагов по времени, источников данных (DS) и архивов (RRA)
  - сами данные, организованные в виде кольцевых архивов (round robin).

## Возможности RRDtool

- Хранение временных рядов
  - Данные лежат в файлах `.rrd` фиксированного размера 
  - В одном файле можно держать сразу несколько метрик
  - Для каждой метрики можно настроить несколько уровней детализации через архивы RRA

- Построение графиков и отчётов
    - Команда `rrdtool graph` строит график по данным из одного или нескольких файлов `.rrd` 

- Обработка данных при построении графика
    - `CDEF` - строит новый ряд как формулу над уже загруженными. Используется, когда нужно преобразовать данные перед рисованием графика, например: сложить два ряда
    - `VDEF` - получает одно агрегированное число для ряда, которое обычно выводится в легенде графика

### В каком случае RRDtool полезен

- Нужно простое и предсказуемое по размеру хранилище метрик
- Нужен простой локальный мониторинг без развёртывания целого стека.
  Например: есть один–два сервера, и нужно смотреть несколько базовых метрик
- Нужно видеть общую картину за долгий период, а не хранить каждое отдельное измерение в исходном виде
- Есть ограничения по ресурсам

### Сравнение графиков 

Отдельной утилиты для сравнения графиков нет, но можно:

- Строить несколько рядов на одном графике
- Строить новый ряд как формулу двух других рядов с помощью `CDEF`
- Получать текстовое сравнение рядов с помощью `VDEF`

## Недостатки

- Невозможно менять шаг или метрики после создания

    Из-за  фиксированной структуры при необходимости сделать другой шаг, другие архивы или дополнительные метрики — старый `.rrd` придётся пересоздавать и мигрировать данные

- Старые данные теряют детализацию или полностью затираются

    Из-за того, что архивы работают как кольцевые буферы детальные данные со временем сворачиваются в более крупные интервалы или просто исчезают. 

- Завязан на инструменты RRDtool и его библиотеки

    Файлы `.rrd` бинарные поэтому, чтобы читать и записывать данные, нужно использовать либо команду `rrdtool`, либо языковые обёртки  

- Отсутствует отдельный интерфейс для построения графиков

- Отсутствует язык запросов

## Требования к ресурсам для работы

Минимальных требований к ресурсам у RRDtool нет. Он рассчитан на работу и на серверах, и на локальных машинах. Ресурсы начинают быть критичны только при очень большом числе метрик/файлов и высокой частоте обновлений.

Для крупных установок используется демон `rrdcached`: он принимает обновления, накапливает их в памяти и периодически записывает на диск пачками. Это снижает нагрузку на дисковую подсистему за счёт дополнительного расхода оперативной памяти под кэш.

## Случаи применения

RRDtool обычно используют в задачах технического мониторинга, где нужно хранить числовые метрики и смотреть графики:

- Мониторинг серверов и сервисов

    Например: загрузка CPU, использование памяти и диска, swap, количество процессов, очереди задач и т.д.

- Мониторинг сетевого оборудования

    Например: пропускная способность интерфейсов, объём входящего/исходящего трафика, ошибки и потери пакетов.

- Сбор показаний датчиков

    Например: температура, влажность, энергопотребление, уровень шума и другие физические величины во времени.

- Встроенный мониторинг в своих утилитах
  
    Например: нагрузка тестового стенда, CI-системы, какого-то конкретного приложения и т.д.
