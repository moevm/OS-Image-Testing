# Collectd

Collectd -- программа-демон, собирающая информацию о производительности системы и приложений и предоставляющая механизмы хранения данных сведений. При этом стоит понимать, что сама утилита отвечает за исключительно за загрузку плагинов, которые и реализуют требуемый функционал.

К преимуществам collectd можно отнести высокую производительность и легковесность (программа написана на языке C, что позволяет ей работать в системах без скриптовых языков или демона cron, например, во встроенных системах).

Поведение программы определяется конфигом, в нем описывается набор плагинов вместе с их возможной спецификацией, указывается временной интервал сбора статистики и устанавливаются дополнительные параметры работы программы.

Пример базового конфига:
```
BaseDir "/var/lib/collectd"
PIDFile "/run/collectd.pid"
Interval 10.0

LoadPlugin cpu
LoadPlugin load

<LoadPlugin df>
  Interval 3600
</LoadPlugin>
<Plugin df>
  ValuesPercentage true
</Plugin>
```

Для работы также требуется определить спецификацию метрик используя *types.db* файл.
Каждая строка файла описывает один тип метрики, который в collectd называется “набором данных“ (“data set”). Каждая строка состоит из двух или более полей, разделенных пробелами и/или горизонтальными табуляциями. Первое поле -- название датасета, второе и последующие -- сама переменная (метрика) или “data source”. Множественные переменные поддерживаются, однако нежелательны.

Поле переменной состоит из имени, типа, наименьшего и наибольшего значения и выглядит следующим образом -- `ds-name:ds-type:min:max`

Тип переменной может быть одним из следующих:
- **GAUGE**     -- Мгновенное значение. Число «как есть». Может расти и падать
- **COUNTER**   -- Счетчик или же постоянно возрастающее число
- **DERIVE**    -- Похож на Counter, но значение может убывать. Используется, когда нужно получить производную (скорость изменения), но источник не является строгим счетчиком
- **ABSOLUTE**  -- Сбрасываемый счетчик. Значение, которое сбрасывается в ноль после каждого чтения. Такое значение хоть и доступно, однако не рекомендовано к использованию.

Также имеется возможность определять собственные типы данных.

## Доступные плагины
Полный список плагинов можно посмотреть в таблице по [ссылке](https://github.com/collectd/collectd/wiki/Table-of-Plugins).

Список содержит более 100 плагинов, так что рассмотрим наиболее распространенные из них.

### Плагины для записи (Output / Write)
Здесь важно отметить, что collectd -- не база данных, а утилита работающая по принципу "собрал данные -- отправил данные". Данные можно как записывать в файлы, так и отправлять в базу данных или на сервер.

Таблица наиболее распространенных плагинов записи метрик:
|Плагин|Описание|
|----|----|
|network|Позволяет отправлять данные на другой сервер collectd или принимать их|
|rrdtool|Записывает данные в локальные RRD файлы|
|csv|Пишет метрики в CSV-файлы|
|write_graphite|Отправляет данные в Graphite (Carbon) по протоколу TCP|
|write_influxdb|Отправляет данные в базу данных InfluxDB|
|write_prometheus|Поднимает веб-сервер, который отдает метрики в формате, понятном Prometheus (scraper)|

### Плагины для измерения параметров
Как уже говорилось ранее, collectd собирает данные по всей системе. Ниже приводится таблица наиболее распространенных плагинов для сбора статистики:

|Плагин|Описание|
|----|----|
|cpu|Собирает статистику использования процессора (User, System, Idle, Wait и т.д.)|
|memory|Мониторит использование оперативной памяти (Free, Used, Cached, Buffered)|
|interface|Считает сетевой трафик (октеты/пакеты) и ошибки на сетевых интерфейсах (eth0, ens33 и т.д.)|
|df|Отслеживает свободное место на примонтированных файловых системах (дисках)|
|disk|Считает операции ввода-вывода (IOPS) и скорость чтения/записи на физических дисках|
|load|Собирает данные о средней загрузке системы (Load Average: 1, 5, 15 мин)|
|swap|Отслеживает использование файла подкачки (Swap)|
|uptime|Фиксирует время работы системы после перезагрузки|
|sensors|Читает данные с физических датчиков (температура CPU, скорость вращения кулеров, вольтаж)|
|processes|Считает количество процессов, сгруппированных по статусу (running, sleeping, zombies)|

### Плагины-обработчики запросов
Плагины "управления" данными:
|Плагин|Описание|
|----|----|
|logfile|Пишет логи работы самого демона collectd (ошибки, предупреждения) в файл|
|syslog|Отправляет логи демона в системный журнал (syslog/journald)|
|threshold|Позволяет задавать пороговые значения (Warning/Failure) и генерировать уведомления при их превышении|
|aggregation|Агрегирует (объединяет) несколько метрик в одну|
|exec|Позволяет запускать ваши собственные скрипты (Bash, Python и др.), забирать их вывод и превращать в метрики|
|unixsock|Создает Unix-сокет, через который можно командой GETVAL запросить текущее значение метрики|

Как таковой обработки в collected не предусмотрено в силу, опять таки, принципа работы. Соответственно нет возможности посчитать среднее значение, вычислять квартилии и проводить прочие статистические операции. По этой же причине отсутствует возможность отправлять запросы -- для этого требуется сторонняя база данных. Единственный вариант -- получение текущего значения через *unixsock*.

Имеется возможность "схлопывать" множество метрик в одну, если использовать *aggregation*. Так, например, можно посчитать среднюю загрузку процессора. Однако перечень операцией ограничивается базовой математикой, а также не представляется возможным генерировать результат, требующий нескольких измерений.

## Применение в контексте тестирования образов
Контекст использования collectd определяется особенностями утилиты, а также принципом ее работы: collectd не хранит и не обрабатывает собираемые данные -- для этого требуется использовать другие программы. Таким образом, collectd можно использовать как промежуточное звено, которое будет собирать необходимые метрики и вносить их в базу данных для дальнейшей обработки и/или анализа в процессе тестирования образа. Высокая скорость выгодно отличает collectd от других ему подобных инструментов.
