# Мониторинг производительности

Мониторинг производительности — регулярный сбор метрик системы и приложений во времени для обнаружения проблем и моделей поведения, проявляющихся со временем. Относится к анализу наблюдений. Его следует использовать на начальном этапе анализа производительности, чтобы быстро получить обзор состояния системы и сузить область поиска.

Состоит из:

1. Списка ключевых метрик для мониторинга по ресурсам
2. Списка категорий детализации метрик (по компонентам/операциям/интерфейсам/процессам), чтобы смотреть значения не только в среднем по системе
3. Списка правил выбора интервала измерения и архивирования
4. Списка правил интерпретации метрик с учётом контекста рабочей нагрузки, требований к задержкам и возможных ограничений ресурсов

Недостатки:

* Выбор интервала измерения и архивирования влияет на обнаружение кратковременных всплесков, при крупных интервалах они могут быть незаметны

* Средние значения могут скрывать выбросы и закономерности: для метрик задержек желательно анализировать распределение и смотреть задержки отдельно по типам операций

* Для некоторых подсистем (особенно файловых систем) в Linux доступно мало встроенных статистик по операциям: из-за этого детализация мониторинга ограничена

* В средах с ограничениями/квотами возможна неверная интерпретация, так как деградация может быть вызвана механизмами регулирования и лимитами

Требуемые вычислительные ресурсы:

* Низкие:
  * Сбор базовых метрик с большим интервалом: фиксируются ключевые показатели (потребление/насыщенность/ошибки/время отклика) без детализации метрик по отдельным категориям и без анализа распределений задержек

* Средние:
  * Более частые измерения и детализация: посекундный сбор предпочтительнее для обнаружения кратковременных всплесков, детализация метрик по отдельным категориям увеличивает объём собираемых данных и нагрузку на хранение и обработку

* Высокие:
  * Сбор и анализ распределений задержек: вместо одного среднего значения требуется исследовать полное распределение, чтобы не пропустить выбросы и закономерности - это увеличивает объём данных и стоимость обработки.
  * Учет механизмов регулирования: при наличии механизмов управления ресурсами дополнительно собирают статистики, показывающие, когда и как срабатывали эти ограничения. Это увеличивает число метрик и объём данных для хранения и обработки

Применимость:
  * Начальный этап анализа производительности

  * При отклонениях метрик от базовой линии для первичной оценки состояния сервиса по данным мониторинга

  * Для постоянного контроля метрик во времени, чтобы видеть изменения и повторяющиеся закономерности в динамике показателей


## Уточнения

Процессор:

* Потребление: величина нагрузки в процентах
* Насыщенность: длина очереди на выполнение или задержка планировщика
* Ограничения: в средах с квотами CPU потребление следует оценивать с учётом этих ограничений
* Интервал измерений: выбор интервала измерения и архивирования влияет на обнаружение кратковременных всплесков, при крупных интервалах они могут быть незаметны, поэтому для их выявления используют метрики насыщенности и при необходимости переходят к более детальному временно́му масштабу анализа

Память:

* Потребление: процент использованной памяти, который можно определить по объёму доступной памяти
* Насыщенность: наличие событий подкачки и сообщений OOM Killer в журнале
* Ограничения: при наличии ограничений/квот памяти нужно собирать статистики, характеризующие наложенные ограничения
* Ошибки: можно выполнять мониторинг ошибок, если есть такая возможность (по аналогии с методом USE), фиксируя их по журналу ядра/системному журналу (сообщения OOM Killer / Out-of-memory) и по логам/метрикам приложений (ошибки выделения памяти)
* Утечки: мониторинг потребления памяти во времени, особенно по процессам, помогает выявить наличие и величину утечек памяти

Файловая система:

* Частота операций: основная характеристика рабочей нагрузки
* Задержки операций: результирующая производительность
* Контекст "норма/ненорма": нормальная задержка зависит от рабочей нагрузки, среды и требований к задержке
* Детализация по типам операций: частоту и задержки можно фиксировать отдельно для чтения, записи, чтобы видеть изменения нагрузки и производительности по конкретным операциям
* Метрики задержек можно фиксировать как среднее в секунду и дополнительно максимум и стандартное отклонение, также необходимо анализировать распределение задержек, чтобы не пропустить выбросы и закономерности
* Упомянутые инструменты:
  * nfsstat(8) - выводит статистику NFS

Диски:

* Потребление диска: доля времени, когда устройство занято обслуживанием запросов ввода-вывода
* Время отклика: время обслуживания запроса ввода-вывода
* Ограничения: узкое место может быть вызвано ограничениями дисковых ресурсов, необходимо собирать статистику, показывающую, как и когда использовались эти ресурсы
* Дополнительные метрики для характеристики нагрузки: IOPS и пропускная способность
* Метрики следует оценивать для каждого диска, чтобы выявлять несбалансированную нагрузку и проблемные диски, также предпочтительно анализировать распределение времени отклика

Сеть:

* Пропускная способность: RX/TX в байтах в секунду
* Соединения: частота создания TCP-соединений в секунду как показатель сетевой нагрузки
* Ошибки: включая счётчики отброшенных пакетов
* Повторные передачи TCP: могут пригодиться для увязывания с проблемами
в работе сети
* Неупорядоченные пакеты TCP: могут вызывать проблемы с производительностью
* Ограничения: в средах с ограничением полосы можно собирать статистики по наложенным ограничениям
