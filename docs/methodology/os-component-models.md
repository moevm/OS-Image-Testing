# Описание аналогов для эмуляции 
### QEMU blkdebug
blkdebug — это встроенный в QEMU механизм основанный на правилах инжекции ошибок на уровне блочного слоя QEMU.
Он подключается как «обёртка» над обычным образом диска и позволяет детерминированно ломать операции чтения/записи. Находится в составе QEMU.

Основной механизм:
- виртуальный диск задаётся как `file=blkdebug:<config>:<image>` — реальный образ `<image>` обслуживается через слой `blkdebug`
- `blkdebug` читает конфигурационный файл правил и применяет их при обработке запросов чтения/записи
- при совпадении правила может быть возвращена ошибка (`EIO`, `ENOSPC` и т.д.) или выполнено управляющее действие, чтобы воспроизвести редкие сценарии 

Какие ошибки можно эмулировать:
- типичные ошибки чтения/записи через `errno`, включая `EIO` и `ENOSPC`
- ошибки в конкретных событиях операций чтения/записи

Количество/частота:
- задаётся правилами и состояниями 

### Linux Device Mapper targets: dm-flakey/dm-delay/dm-dust
Device Mapper — подсистема ядра Linux, позволяющая строить виртуальные блочные устройства `/dev/mapper/<name>` поверх реальных дисков и модифицировать поведение операций чтения/записи. Для отказных сценариев полезны цели `dm-flakey`, `dm-delay`, `dm-dust`.  Находится в ядре Linux.

**dm-flakey**:
- работает циклами «норма/ненорма»: некоторое время устройство доступно, затем некоторое время ведёт себя ненадёжно, и цикл повторяется
- в плохой фазе можно выбрать поведение: возвращать ошибки, дропать записи или портить данные 

**dm-delay**:
- задержки можно задавать раздельно для чтения и записи
- добавляет настраиваемую задержку чтения/записи/flush 
- применяется как самостоятельный инструмент или в паре с `dm-flakey`, чтобы получить «и ошибки, и тормоза».

**dm-dust**:
- эмулирует плохие сектора: позволяет указать произвольные блоки, где чтения начнут падать, и включать/выключать режим отказов в произвольный момент

Количество/частота:
- у `dm-flakey` - через длительности фаз «норма/ненорма» и/или вероятность порчи и режимы drop/error/corrupt
- у `dm-dust` — через список «плохих блоков» + управление в нужный момент

### nbdkit
`nbdkit` поднимает блочное устройство по протоколу Network Block Device, а клиент (хост или VM) подключает его как `/dev/nbdX`. `nbdkit` - отдельный userspace‑инструмент, ставится из пакетов дистрибутива

Основные фильтры для отказов:
- `nbdkit-error-filter` - инжектит случайные ошибки в ответы сервера; частота задаётся `error-rate` 
- `error-file` - позволяет включать инжекцию только, когда существует файл
- `nbdkit-delay-filter` - добавляет задержки чтения/записи 
- `nbdkit-rate-filter` - ограничивает пропускную способность 
- `evil filter` - добавляет коррупцию данных без ошибок чтения/записи

Количество/частота:
- ошибки: `error-rate`
- задержки/лимиты: параметрами delay/rate‑фильтров
- corruption: настраиваемыми режимами/интенсивностью

### Linux fault injection
В ядре Linux есть инфраструктура fault-injection, позволяющая искусственно ломать различные подсистемы (в том числе блочный ввод-вывод) с управляемой частотой. Используется для тестирования того, как ОС обрабатывают ошибки. Находится в ядре Linux, управление обычно выполняется через debugfs в /sys/kernel/debug/....

Основной механизм:

- В ядре есть точки инжекции, где можно заставить операцию завершиться ошибкой или повести себя нестандартно
- Параметры задаются через файлы в debugfs, после чего ядро начинает инжектить сбои по заданным правилам.

Ключевые варианты для дискового чтения/записи:

- fail_make_request — инжекция ошибок на уровне block layer: позволяет заставить операции чтения/записи завершаться ошибкой на пути обработки запросов
- NVMe Fault Injection: через /sys/kernel/debug/nvme*/fault_inject можно инжектить NVMe status codes и флаг no_retry, чтобы эмулировать поведение контроллера NVMe 
- для NVMe — параметры в `fault_inject` для статуса и no_retry 

Какие ошибки можно эмулировать:

- Ошибки выполнения операций чтения/записи на уровне block layer 
- Специфичные NVMe-ошибки

### Linux MD FAULTY
В Linux software RAID предусмотрен режим FAULTY, предназначенный для симуляции отказов, чтобы тестировать файловые системы и уровни RAID. Находится в ядре Linux, управление — через `mdadm`.

Основной механизм:
- Создаётся `/dev/mdX`, который работает как блочное устройство, но по правилам начинает возвращать ошибки чтения и/или записи
- Можно выбрать тип отказа для чтения и записи (режимы задаются при настройке массива).

Типы отказов:
- transient - сбой возникает “однократно/периодически”, и следующая попытка может пройти успешно.
- persistent - при обращении к данному месту ошибка повторяется стабильно.
- fixable - ошибка чтения держится, пока в этот же адрес не произойдёт запись.

Количество/частота:

- Управляется выбором режима (transient/persistent/fixable) отдельно для read/write;
- mdadm позволяет выбирать комбинации вида “write-transient”, “read-persistent”, “read-fixable” и т.п. — задается какой тип операции ломает и как.

### Mininet
Mininet — это сетевой эмулятор, который позволяет на одной машине поднять виртуальную сеть и запускать внутри неё реальные приложения и сетевые протоколы, чтобы воспроизводимо проверять их поведение на разных топологиях и условиях канала.

Механизм работы заключается в том, что Mininet создаёт виртуальные узлы и соединяет их виртуальными ссылками, а параметры каждого линка можно задавать так, чтобы узлы видели канал как медленный, узкий или с потерями.

Какие ошибки/ухудшения эмулирует:
- задержка
- потери пакетов
- ограничение пропускной способности
- ограничение очереди

### Mininet-WiFi
Mininet-WiFi — это расширение Mininet, которое превращает его в эмулятор Wi-Fi-сети: добавляет виртуальные Wi-Fi станции и точки доступа, чтобы тестировать сценарии беспроводной сети.

Механизм работы заключается в том, что к обычной логике Mininet добавляются виртуальные Wi-Fi устройства, работающие на стандартных Linux драйверах, и появляется возможность задавать свойства мобильной станции — позицию и движение относительно точек доступа. За счёт этого можно проигрывать сценарии: приближение/удаление от AP, смена точки доступа, деградации связи по мере движения.

Какие ошибки/ухудшения эмулирует:
- деградацию качества беспроводного канала по мере изменения положения
- разрывы/нестабильность соединения при движении
- смену точки доступа

### CORE 
CORE — это сетевой эмулятор для построения виртуальных сетей, который работает в реальном времени и позволяет запускать в узлах настоящие приложения и протоколы, как на реальных машинах.

Механизм работы заключается в том, что пользователь собирает топологию, CORE поднимает лёгкие виртуальные узлы и соединяет их, после чего сеть начинает жить как настоящий стенд: можно запускать сервисы, маршрутизацию, диагностику, и при этом менять топологию/состояния линков. CORE также умеет подключать эмулируемую сеть к физической.

Какие ошибки/ухудшения эмулирует:
- разрывы связности 
- отказ/перезапуск узлов в топологии как управляемое событие

### tc netem 
tc netem — это встроенный в Linux механизм сетевой эмуляции в подсистеме Traffic Control, который позволяет искусственно ухудшать работу сети и тем самым воспроизводимо проверять устойчивость протоколов и распределённых приложений.

Механизм работы заключается в том, что для выбранного сетевого интерфейса включается режим сетевой эмуляции: все исходящие пакеты начинают проходить через слой управления очередью перед отправкой на устройство. На этом этапе пакеты могут удерживаться во внутренней очереди и выпускаться по таймеру, либо с заданной вероятностью подвергаться ухудшениям — отбрасываться, дублироваться, приходить в изменённом виде или выходить в другом порядке. Ограничение пропускной способности обычно настраивают совместно с механизмами Traffic Control для шейпинга скорости, чтобы смоделировать узкий канал.

netem по умолчанию работает на исходящем трафике. Если нужно портить входящий, часто используют зеркалирование.

Какие ошибки эмулирует:
- задержка
- потери пакетов
- дублирование пакетов
- порча пакетов 
- перестановка пакетов 
- ограничение пропускной способности

### Toxiproxy 
Toxiproxy — это пользовательский TCP-прокси для сетевой эмуляции, который ставят между клиентом и реальным сервисом, чтобы воспроизводимо проверять, как приложение переживает деградации сети: медленный канал, нестабильные соединения, разрывы и зависания.

Механизм работы устроен так: toxiproxy поднимает отдельную точку подключения (proxy endpoint). Клиент подключается не напрямую к серверу, а к прокси-адресу, а прокси устанавливает соединение с сервером назначения. После этого в прокси можно включать “toxics” — набор правил, которые вмешиваются в проходящий трафик и искусственно ухудшают его поведение. Правила могут добавлять задержку к передаче данных, ограничивать скорость, провоцировать ситуации с таймаутами за счёт удержания потока, а также имитировать аварийное состояние канала, когда соединение принудительно разрывается или становится недоступным.

Какие ошибки/сбои эмулирует:
- Высокая задержка
- Ограничение пропускной способности
- Таймауты и зависания 
- Принудительный разрыв/недоступность канала (модель outage).

### cgroups 
CPU-контроллер cgroups — это механизм ядра Linux для управления вычислительными ресурсами групп процессов. В контексте тестирования его используют как программную модель перегруженного процессора: приложение получает меньше процессорного времени, чем обычно, и начинает вести себя так, как будто CPU слабее, занято соседями или работает в режиме жёстких ограничений. Groups CPU моделирует ограничение планировщика.

Механизм работы заключается в том, что процессы объединяются в cgroup, и для этой группы задаётся политика распределения CPU-времени. Для обычных задач планировщика можно указать квоту — сколько времени на CPU группа имеет право потребить в течение фиксированного периода. Пока квота не исчерпана, задачи выполняются нормально. Как только лимит выбран, ядро начинает душить группу: процессы принудительно приостанавливаются с точки зрения планировщика и ждут начала следующего периода, когда квота снова станет доступна.

Какие ошибки/сбои эмулирует:
- Деградацию производительности: рост времени отклика, падение throughput
- Срыв дедлайнов
- Starvation фоновых задач

### gem5
gem5 — это архитектурный симулятор, который моделирует железо от CPU до устройств ввода-вывода и позволяет грузить ОС и запускать бенчмарки, но уже с упором на детальную модель микроархитектуры.

Механизм работы заключается в том, что пользователь собирает конфигурацию виртуального компьютера (тип CPU, кэши, память, устройства), дает образ диска с ОС, и gem5 воспроизводит выполнение системы в симуляции. 

Какие ошибки/сбои эмулирует:
- разные модели CPU, переключение режимов
- кэш/память/когерентность как управляемые параметры

Какое ОС ставится:
- Full System (поддерживаются разные ISA)
- Syscall emulation режим: запускаются пользовательские программы, а системные вызовы эмулируются gem5
- ОС, которая умеет работать с тем набором устройств, которые есть в gem5

Какие устройства эмулирует:
- CPU, MMU, кэши, шины
- Контроллеры памяти и модели DRAM, адресное пространство.
- Контроллеры прерываний, таймеры, системные регистры 
- UART/serial console, RTC и тд
- Простые модели Ethernet/NIC 
- Модель диска и контроллеры хранения 

Отличие от ВМ:
- Код выполняется в симуляции, поэтому сильно медленнее, чем на ВМ
- Можно задавать и менять модель CPU, размеры/политики кэшей, межсоединения, модель памяти/DRAM, получать подробные статистики
- Удобно менять архитектурные параметры и смотреть, как ОС реагирует на другую архитектуру, тогда как в ВМ их не поменяешь
- В gem5 возможен fast-forward через KVMCPU (ускорение до близкого к нативному при наличии KVM), но в этом режиме gem5 не собирает полноценные архитектурные статистики, поэтому его используют для разгона или загрузки, а затем переключаются на детальный CPU для измерений

Нюансы работы с Full System:
- В режиме Full System gem5 использует disk image с установленной ОС; поддерживаются только raw-образы, поэтому образы в других форматах нужно предварительно конвертировать в raw
- По умолчанию изменения в файловой системе гостя не сохраняются в исходный образ: используется COW-слой, который сбрасывается после завершения симуляции (при необходимости COW можно отключить)
- Загрузка обычно происходит без bootloader: gem5 часто сам загружает kernel в память, поэтому kernel задаётся отдельно от disk image и легко заменяется без изменения образа

### DRAMSim3
DRAMSim3 — это симулятор подсистемы памяти, который моделирует тайминги DRAM и поведение контроллера памяти для разных стандартов (DDR/LPDDR/GDDR/HBM и др.).

Механизм работы заключается в том, что DRAMSim3 получает запросы к памяти из трасс или из CPU-симулятора и рассчитывает, как они обслуживаются с учётом очередей команд, банков, задержек и ограничений протокола. В результате можно увидеть, где памяти не хватает и как меняются задержки при разных настройках.

Какие ошибки/сбои эмулирует:
- рост латентности из-за таймингов/конфликтов очередей команд
- разные типы DRAM и их влияние на задержки и пропускную способность
- узкие места контроллера

### GPGPU-Sim
GPGPU-Sim — это детальный симулятор GPU, который моделирует современные NVIDIA-подобные GPU при выполнении CUDA/OpenCL нагрузок и даёт метрики производительности.

Механизм работы заключается в том, что пользователь собирает приложение под прогон в симуляторе, и GPGPU-Sim воспроизводит выполнение на модели GPU и считает, где тратятся циклы и почему.

Какие ошибки/сбои эмулирует:
- влияние GPU на задержки
- эффекты разветвления с расхождением потоков и планирования блоков
- поведение GPU-микроархитектуры на разных типах кернелов 

# Сценарии тестирования

### Диск 

- Рост времени отклика — добавление задержки чтения/записи, чтобы смоделировать долгие операции чтения/записи.
Инструменты: dm-delay, nbdkit delay-filter

- Ограничение пропускной способности — ограничение скорости передачи данных или косвенное снижение пропускной способности за счёт шейпинга.
Инструменты: nbdkit rate-filter, для dm-delay ограничение достигается косвенно через увеличенную задержку.

- Ошибки чтения/записи — возврат ошибок при чтении/записи для проверки обработки EIO/ENOSPC и подобных событий.
Инструменты: QEMU blkdebug, nbdkit error-filter, dm-flakey, Linux fault injection.

- Плохие блоки/сектора — ошибки при обращении к заданным блокам, моделирование дефектов носителя.
Инструменты: dm-dust.

- Коррупция данных без явной ошибки — порча данных таким образом, что операция формально проходит, но данные искажаются.
Инструменты: nbdkit evil filter, dm-flakey.

- Отказы на уровне слоёв хранения — проверка поведения надстройки при ошибках чтения/записи.
Инструменты: Linux MD FAULTY.

### Сеть

- Деградация канала — добавление задержки, потерь, перестановки, дублирования пакетов, ограничение пропускной способности.
Инструменты: tc netem.

- Недоступность сервиса / разрыв соединения между компонентами — ухудшение связи между конкретным клиентом и сервером.
Инструменты: Toxiproxy.

- Топологии и сетевые разделения — сценарии с несколькими узлами и управляемой связностью, включая разрыв сети между сегментами.
Инструменты: Mininet, CORE.

### Процессор

- Ограничение вычислительных ресурсов — уменьшение доступного процессорного времени для группы процессов.
Инструменты: cgroups CPU.
Эффект: рост времени отклика, падение пропускной способности, возможные таймауты сервисов и срыв дедлайнов.

### Память и микроархитектура

- Высокая латентность памяти — моделирование деградации подсистемы DRAM.
Инструменты: DRAMSim3.

- Другая архитектура системы — моделирование поведения CPU/кэшей/памяти/устройств на уровне архитектурного симулятора для исследовательского анализа.
Инструменты: gem5.
Эффект: можно воспроизводимо сравнить чувствительность дистрибутива к параметрам кэшей/памяти и конфигурации платформы.

### GPU

- Другая модель GPU — моделирование производительности другой видеокарты или более слабой конфигурации.
Инструменты: GPGPU-Sim.

- медленная видеопамять — моделирование влияния памяти GPU на производительность.
Инструменты: GPGPU-Sim.

- Конкуренция за GPU — запуск параллельной фоновой GPU-нагрузки для проверки поведения приложения при совместном использовании ускорителя.
Эффект: рост времени выполнения, падение пропускной способность.

- Недоступность GPU — запуск приложения без доступа к GPU для проверки корректного запасного варианта на CPU и корректной обработки ошибок ускорителя.
Эффект: отсутствие падений/зависаний, предсказуемый переход на CPU.
