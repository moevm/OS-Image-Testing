# Метод оценки размера рабочего набора (working set size, WSS)

**WSS** - размер рабочего набора (объём часто используемой памяти), сколько памяти необходимо приложению для работы.

Время на реализацию:
* Вручную: быстро. Нужно экспериментально получить примерный размер путём оценивания изменений в выводе утилит.
* Автоматизация: некоторые инструменты дают готовую грубую оценку WSS.

Накладные расходы:
* Основные расходы связаны со сбором данных.
* Готовые решения для оценки WSS могут увеличить latency процесса на `10%`.

## Подходы к оценке

### Наблюдение за метриками paging/swapping

Идея подхода в следующем:

* Если WSS > RAM (высокая нагрузка), то рабочий набор не помещается в физическую память и в метриках можно наблюдать устойчивую подкачку.
* Если WSS == RAM, то рабочий набор почти полностью занимает всю доступную память, в метриках нет подкачки, но память активно сканируется на предмет редко используемых данных.
* Если WSS < RAM, то рабочий набор меньше, чем объём физической памяти, нет ни подкачки, ни сканирования памяти.

`vmstat 1` - показывает нужные метрики (подкачка) в выводе: `si` и `so`

### Наблюдение за метриками производительности (PMC)

Суть подхода в косвенной оценки WSS по соотношению hit/miss у кэшей.

```
LLC% = 99.85%  # L3 кэш
Вывод: WSS ≤ размер LLC (например 24 МБ)

В реальности WSS = 10 МБ
```

Подход работает только если данные хорошо кэшируются, а результат может зависеть от внешних факторов (измерения в кэш-линиях, включение шаблона доступа и тд).

Возможно провести быструю оценку без накладных расходов.

[Готовые инструменты:](https://github.com/brendangregg/pmc-cloud-tools)

`pmcarch` - выводит информацию о количестве циклов и инструкций CPU, LCC

`cpucache` - выводит информацию о L1, L2, LCC

### Оценка через бит доступа страниц

Для всех рассматриваемых страниц (на них делится память) бит доступа ставится 0. Если страница задействуется, то он заменяется на 1.

Соответственно спустя определённое время можно повторно замерить биты и оценить величину WSS для разных видов памяти.

```
1 мс ~ L1/L2
10 мс ~ L3
100 мс ~ ОЗУ
5 с ~ WSS
30 с ~ почти всё, что используется
```

[Готовые инструменты](https://github.com/brendangregg/wss):

`wss.pl` - реализация оценки через бит доступа, принимает PID и время. Ref - оценка WSS

```
# ./wss.pl 423 0.1
Watching PID 423 page references during 0.1 seconds...
Est(s)     RSS(MB)    PSS(MB)    Ref(MB)
0.107       403.66     400.59      28.02
```

`wss-v1` - более безопасный аналог, так как использует специальные флаги. Эффективен для малых размеров данных

`wss-v2` - аналогичная утилита, но эффективна уже для больших по объёму процессов

Накладные расходы всех перечисленных инструментов могут замедлить работу приложения до `10%`.

Если в странице задействован даже 1 байт, то она вся считается задействованной, что может влиять на итоговую оценку.

### Уменьшение потребления памяти

Суть подхода заключается в уменьшении объёма доступной памяти приложения во время его работы и измерении его производительности. Когда производительность начинает резко падать, а подкачка увеличиваться, то WSS перестаёт умещаться в доступной памяти.

Этот подход не рекомендуется к применению в промышленном окружении, поскольку в нём намеренно снижается производительность программы.
