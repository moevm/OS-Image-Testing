# Утилиты для performance тестирования 

## Какие метрики можно анализировать

По требованиям необходимо тестировать сетевую и дисковую подсистемы. Возможно анализировать несколько метрик.

Связанные с дисковой подсистемой:
* IOPS (Input/Output Operations Per Second) - количество операций чтения/записи в секунду
* Latency - время выполнения запроса
* Queue Depth - количество операций ввода-вывода, ожидающих выполнения

Связанные с сетевой подсистемой:
* Network Throughput - пропускная способность, скорость передачи данных по сети
* Packet Latency - время передачи пакета от отправителя к получателю
* Packet Loss - процент потерянных пакетов при передаче данных
* Jitter - вариации задержки пакетов данных при их передаче по сети. Измеряется в миллисекундах (мс) и отражает изменчивость времени задержки пакетов данных при их передаче между двумя точками сети


## Утилиты 


Для метрик дисковой системы:
1. FIO (Flexible I\O Tester)
   * Метрики: IOPS, Throughput, Latency, Queue Depth
   * Создает потоки ввода/вывода данных с диска, чтобы оценить производительность файловой системы Linux
   * Утилита посчитает IOPS и пропускную способность системы, а также позволит оценить глубину очереди операций ввода/вывода. Утилита работает со специальными файлами, в которых прописываются настройки и условия теста. 
   * FIO агрегирует данные, суммируя результаты из нескольких потоков, что позволяет получить общую картину производительности системы.
   * Пример содержимого файла 
   
     ```
     [global]
          name=fio-rand-read
          filename=fio-rand-read
          rw=randread
          bs=4K
          direct=0
          numjobs=1
          time_based=1
          runtime=900
     ```
     
     Этот конфигурационный файл задаёт тест, который в течение 15 минут выполняет случайные операции чтения блоками по 4 КБ из файла fio-rand-read одним потоком, используя не прямой ввод-вывод (кэш ОС). 
   * Может быть добавлена в образ Yocto через bitbake или установлен на целевую систему после запуска
2. ioping
   * Метрики: latency
   * Простой инструмент для мониторинга задержек ввода-вывода диска в режиме реального времени
   * ioping выводит статистику по задержкам, позволяя оценить среднюю и максимальную задержку
   * Параметры:
     * `-c count` (остановить после указанного количества запросов)
     * `-w deadline` (остановить после указанного количества времени)
     * `-p period` (отображать raw статистику после каждого указанного количества запросов)
     * `-P period` (отображать raw статистику после каждого указанного количества секунд)
     * `-i interval` (интервал между запросами в секундах)
     * `-s size` (размер запроса)
     * `-S size` (размер рабочего набора)
   * Может быть добавлена в образ Yocto через bitbake

Связанные с сетью:
1. iperf
   * Метрики: Network Throughput, Packet Latency, Packet Loss (UDP), Jitter (UDP)
   * Инструмент для тестирования сети, который может создавать потоки данных TCP и UDP и измерять пропускную способность сети, по которой они передаются. Для UDP тестов можно оценить packet loss.
   * Может быть добавлена в образ Yocto через bitbake
   * iperf агрегирует данные, вычисляя среднюю пропускную способность и процент потерянных пакетов за тестовый период
   * Параметры:
     * `-s` (Запускает утилиту в режиме сервера, ожидая подключения клиента)
     * `-c IP` (Запускает утилиту в режиме клиента, подключаясь к серверу по указанному IP)
     * `-u` (Включает тестирование UDP вместо TCP)
     * `-P` (Указывает количество параллельных потоков для тестирования)
     * `-t` (Устанавливает длительность теста в секундах)
   * Вывод включает Bandwidth (Пропускная способность (в Mbps)), Packet Loss (Процент потерянных пакетов), Jitter (Вариации задержки (в мс))
     
     Пример вывода
   
     ```
      [ ID] Interval       Transfer     Bandwidth       Jitter   Lost/Total Datagrams
      [  3] 0.0-10.0 sec   1.25 MBytes   1.05 Mbits/sec   0.123 ms  5/1000 (0.5%)
     ```
2. hping
   * Метрики: Packet Latency, Packet Loss
   * Утилита для генерации и анализа сетевых пакетов на основе TCP/IP. Она позволяет отправлять кастомизированные пакеты (ICMP, TCP, UDP) и анализировать их отклик
   * Может быть добавлена в образ Yocto через bitbake
   * Параметры:
     * `--icmp` (Отправка ICMP эхо-запросов)
     * `--udp` (Отправка UDP пакетов)
     * `--interval` (Интервал между отправкой пакетов)
     * `--count число (-c)` (Количество отправляемых пакетов)
     * `--fast` (Отправка 10 пакетов в секунду)
     * `--flood` (Максимальная скорость отправки пакетов)
   * Вывод включает статистику отправленных и полученных пакетов, а также время отклика (RTT). Для оценки джиттера необходимо вручную анализировать разницу между временем отклика каждого пакета.


## Сбор данных в течение длительного времени 

### Вариант 1 - простой

Инструменты:
* Утилиты для тестирования (ipref, fio и др)
* Для настройки регулярности запуска тестов использовать, например, cron
* Визуализация данных с помощью matplot, gnuplot

План работы:
1. Запуск Yocto-образов в QEMU с доступом по SSH
2. Создание и запуск скриптов для тестирования на целевой машине
3. Настройка периодичности запуска тестов
4. Визуализация данных на основе полученных данных

### Вариант 2 - более сложный

Инструменты:
* Prometheus - система мониторинга, которая собирает метрики из различных источников и хранит их в базе данных
* Grafana - инструмент визуализации данных, который позволяет создавать графики и панели на основе данных из Prometheus
* Ansible - инструмент для автоматизации конфигурации и запуска тестов
* Утилиты для тестирования (ipref, fio и др)

План работы:
1. Запуск Yocto-образов в QEMU с доступом по SSH
2. Установка, настройка Prometheus для сбора метрик с целевой системы
3. Настройка экспортеров для утилит. Для iperf существует специальный экспортер для iperf3, который позволяет отправлять метрики в Prometheus. Для fio, hping, ioping не существуют готовые экспортеры, поэтому вам нужно будет написать скрипты, которые будут запускать эти утилиты, парсить результаты и отправлять их в Prometheus через Pushgateway. Также можно и для iperf. Pushgateway — это компонент экосистемы Prometheus, который позволяет отправлять метрики в Prometheus из скриптов или утилит, не поддерживающих протокол Prometheus по умолчанию
4. Настройка Grafana, создайте панели для визуализации данных из Prometheus
5. Автоматизация тестов с помощью Ansible или скриптов
6. Настройка cron для периодического запуска тестов
